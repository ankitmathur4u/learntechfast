A lot of misconception lies about what is "Data Engineering" and what problem does it solve.

To clear out the doubt- it does not overshadow any of the existing technology (especially any programming language) in fact it solves a problem that was not solved earlier.

Data Engineering brings all the data which is relevant to address a business problem from the history to the last-second transaction on the same page. Not just that, it brings all data in a neat, clean, and analytical manner. It helps the business teams to make data-driven decisions.

Data engineering begins with the automation of extraction of data (at source), the transformation of data (on the way), and Loading (the data at the destination). And, to access the data for the end-user there are BI tools. To make it user-friendly there are graphs and maps.

With the help of data engineering, all the relevant departments get the information they need from a 360-degree perspective. For instance, a marketing team can now see the sales funnel. A product management team can see the cost benefits. The management can see the high-level KPI of the business. 

This goes on well for some time and all departments get their share of benefit until the volume of data increases over a period of time. Now, this new “big data” makes the queries, dashboards, and graphs slow to respond and load for users. Similarly the pipeline- which was responsible for bringing the data after ETL become slow and impact the businesses.

So you have another problem to solve called- “Big Data”.

Data engineering is backed by “Big Data” which addresses the problem of - Volume, Velocity, Variety, Veracity.

 

Now, that we have a lot of data. We need a lot of computing power, which cannot be forced upon a single computer, and hence we need a concept of “distributed computing”. This means a lot of commodity hardware. These computers are scalable - which means we can add or remove them quickly whenever needed. Finally, you need a framework that can run the job on this distributed computing environment on the data which is usually of hundreds of GB and sometimes of TBs as well.

Now what you see is Data warehouse (DHW), consolidation of data from multiple sources- sales, traffic data, demographic data, the meaningful joining of data. DWH has complex analytic queries. 

This solves your time problem in the ETL pipeline.

Now, another twist comes when Data scientists join the team and they need data to predict the sales/services. They need the data in close to raw format as well as the data which is streaming into the system. Hence we need “Data lakes”.

Since the data lake has the data in near raw format. This follows the concept of ELT - Extract, Load, and then transform only when and as needed. 

We expect the latest data to be provided to the end-user using streaming data, i.e. real-time data. The best way to provide real-time data to users is using an asynchronous model, one such model is a publisher-subscriber model. Here we decouple the source of data from the consumer of data. Kafka is the most popular pub-sub model.

Maintenance of this DWH and Data Lakes, introducing new data sources, variation in the transformation of data for different use cases, addressing performance issues are some of the ongoing and business usual challenges that a Data Engineer resolves.
